# 7. 节点画布是观察窗，不是编辑器

日期：2026-07-04

## 状态

已接受

## 背景

Workflow 需要 Dify 式的节点画布呈现（有节点、有流动感）。Dify 的画布是编辑器：
节点是确定性积木（HTTP/分支/LLM 调用），拖拽即可组装出可执行流程。Raven 的
节点本质是「一段带意图的 agent 委托 + 人拍板点」，价值在 prompt、合同约束和
返工循环里——这些由 `defineWorkflow` 代码表达（ADR-0003）。若画布做成编辑器，
需要发明一套无代码可执行的节点 DSL，且图编辑产物（数据库记录）会与既有的
「沉淀以 PR 产出、review 合并生效」机制形成两套真相源。

## 决定

画布只用来"看"：设计态渲染合同的阶段/门禁拓扑；运行态在同一张图上呈现流动
（当前节点、完成节点的产出摘要、等待人拍板的节点高亮）。创建与修改 workflow
不走拖拽，走对话式创建（agent 起草 defineWorkflow 文件 → PR review 合并即注册）
与沉淀钩子。

## 后果

- 工程量投向运行态流动感与决策体验，而非图编辑 DSL。
- 编辑门槛（对话 + PR）本身构成"宁缺毋滥"的质量闸门。
- 合同仍是唯一真相源；画布是合同与运行数据的纯投影，无独立可变状态。
- 运行态画布依赖阶段级进度上报（SDK 需新增阶段进度原语，另行设计）。
