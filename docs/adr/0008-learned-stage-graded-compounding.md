# 8. 沉淀阶段激活 Learned 尾巴，分级触发深挖

日期：2026-07-04

## 状态

已接受

## 背景

生命周期的 Merged → Observed → Learned 段一直是死状态，而沉淀（workflow /
skill / 事实与口径）没有制度化的宿主：升轨提议曾对任意完成的裸交付触发，
违反"宁缺毋滥"。全量对每条需求做 agent 复盘则把 token 花在大量"无可沉淀"
的空转上。

## 决定

1. 沉淀成为生命周期一等阶段：需求走到 Learned 才算真正关闭。
2. 三类资产三档门槛：事实与口径逐条沉淀（人确认生效）；skill 单次交付即可
   提议（PR 生效）；workflow 必须同构交付重复 N 次（N=3）才提议（PR 生效）。
3. 分级触发成本模型：Learned 时先做零成本结构化归档（轨迹特征 + 同构计数，
   纯代码）；agent 深挖仅由三种信号触发——同构计数达标、轨迹强信号（门禁
   驳回/返工/执行自报）、用户手动"沉淀这条"。
4. 主进料口是执行现场自报：SDK 新增 `ctx.learning()` 原语，执行中的 agent
   随手记录值得沉淀的心得，边际成本为零；事后深挖只是补充。

## 后果

- "宁缺毋滥"同时约束资产质量与 token 开销。
- 自进化回路闭合：资产改善执行，执行自报喂沉淀，沉淀更新资产。
- 同构计数是对 Learned 归档的查询，不另建系统。
- 需要 SDK/控制面支持 learning 记录与轨迹特征归档（实现另行设计）。
