1. 核心判断
   智能研发平台要解决的不是“再给工程师一个 AI 编码工具”，而是把业务型应用软件的研发过程改造成可持续运行的 三层 loop 系统。
   Andrew Ng 把 0 到 1 产品构建拆成三类循环：
1. Agentic coding loop：给定规格和验证方式，Agent 自己写代码、跑测试、修复失败，直到满足规格。
2. Developer feedback loop：人查看当前产品，基于产品判断、设计品味和上下文优势，继续修正规格和方向。
3. External feedback loop：通过用户反馈、生产数据、A/B 实验、竞品分析和运营信号，反过来影响产品判断。
   这三个循环的时间尺度不同：

   因此，智能研发平台的产品定位应调整为：
   面向业务型应用软件的 AI Loop Platform：用本地工作台承载人和本地 Agent 协作，用云端执行环境承载长程 Agent 和外部反馈自动化，用任务编排系统把三层循环连接起来。
2. 当前问题
   现在 AI 编码能力已经足够强，但组织级研发仍然卡在三个地方。
   第一，coding loop 仍然太依赖人盯会话。工程师在多个终端、多个 worktree、多个 PR、多个 Agent 会话之间切换，实际瓶颈从“模型会不会写代码”变成“人能不能管理这么多 Agent”。
   第二，developer feedback loop 缺少工作界面。人看到实现结果之后，会改变产品判断、设计取舍和规格细节，但这些反馈通常散在 IM、文档、口头讨论和 PR comment 里，不能稳定变成下一轮 Agent 可执行的规格。
   第三，external feedback loop 还没有自动化。上线后的用户行为、业务单据、交易流水、审批记录、异常事件、A/B 结果、客户反馈和竞品变化，往往停留在数据平台、运营报表和会议纪要中，不能自动驱动下一轮研发任务。
   第四，任务编排还停留在人管人/人管 Agent。任务、会话、PR、环境、运行日志、阻塞项和验证证据没有统一状态模型，导致 Agent 多了以后反而增加管理成本。
3. 产品目标
   平台要同时建设两个工作界面和两个底座：本地 Agent 工作台、云端 Loop 控制台、云端执行环境和多 Agent 编排系统。
   3.1 本地 Agent 工作台
   对标 Orca 这类本地 Agent Development Environment。
   目标是给工程师、产品负责人和技术负责人一个本地工作界面，用来操作本地仓库、本地 Agent、本地 worktree，同时查看云端任务状态。
   核心作用：
- 管理本地多个 coding agent 会话。
- 每个任务使用独立 worktree，避免文件冲突和上下文污染。
- 查看 diff、预览页面、运行测试、检查 CI、批注问题并发回 Agent。
- 支持 Claude Code、Codex、Cursor CLI、Gemini、OpenCode 等不同 CLI Agent。
- 查看云端编排系统中的任务状态、阻塞项、PR、验证证据和回流信号。
  本地 Agent 工作台主要服务 Agentic coding loop 和 Developer feedback loop。
  3.2 云端 Loop 控制台
  对标 Multica、OpenAI Symphony 的产品界面方向。
  目标是提供一个给人使用的云端工作界面，用来查看任务、审查 Agent 产出、处理阻塞、审批关键节点，并把外部反馈转成下一轮产品和工程输入。
  它回答的问题是：人如何看见、判断和干预这些 loop？
  核心作用：
- 展示任务、Agent、PR、预览、验证证据和阻塞状态。
- 展示外部反馈、生产数据、A/B 结果和用户反馈分析。
- 支持人审查 Agent 产出，并决定继续迭代、合并、上线或废弃。
- 支持人把反馈改写成产品假设、规格更新、工程任务或验证任务。
- 支持人处理审批、风险确认、优先级调整和任务分派。
  它本身不负责执行任务，也不直接决定调度策略；它是人类进入三层 loop 的操作台和观察窗。
  云端 Loop 控制台主要服务 Developer feedback loop 和 External feedback loop。
  3.3 云端执行环境
  对标 Trigger.dev 的持久任务运行环境。
  它不是普通 CI，也不是简单 serverless，而是面向长程 Agent 和后台 workflow 的执行层。
  核心能力：
- 长任务无固定超时。
- 持久队列、并发控制、重试和幂等。
- cron、Webhook、事件触发。
- 人类审批等待点。
- Python、浏览器自动化、爬虫、数据分析、LLM 评估任务。
- 实时日志、trace、成本、失败原因和任务状态监控。
  它负责把第三层 External feedback loop 自动化。
  3.4 多 Agent 编排系统
  对标 OpenAI Symphony 的后台编排思想。
  它不是给人看的主界面，而是驱动任务流转和 Agent 分配的控制引擎。
  它回答的问题是：系统如何自动推进这些 loop？
  平台不应让人去微观管理每个 Agent 会话，而应让任务系统成为控制面：
- 任务进入看板。
- 编排器判断任务类型、依赖和优先级。
- 分配给本地 Agent、云端 Agent 或 Agent squad。
- 创建独立工作空间。
- 跟踪执行状态、阻塞项、PR、验证证据。
- Agent 卡住或失败时重试、换 Agent、升级给人。
- 任务完成后进入人工审查或自动进入下一轮 loop。
  它负责把三层 loop 串起来，并向云端 Loop 控制台提供状态、证据、阻塞和审批点。
  3.5 云端 Loop 控制台与多 Agent 编排系统的关系
  云端 Loop 控制台和多 Agent 编排系统是一前一后两层。

4. 三层 Loop 设计
   4.1 第一层：Agentic Coding Loop
   这是内层工程循环，目标是让 Agent 在明确规格和验证条件下自动完成实现。
   输入：
- 产品规格
- 任务边界
- 仓库上下文
- 代码规范
- 测试与验收标准
- 可运行环境
  执行过程：
1. Agent 读取任务和上下文。
2. 创建或进入独立 worktree / sandbox。
3. 拆解实现计划。
4. 修改代码、补测试、更新文档。
5. 运行 test、lint、type check、contract test、端到端检查。
6. 失败则继续修复。
7. 成功后生成 PR、验证证据和说明。
   平台能力：
- 本地工作台发起和观察。
- 云端执行环境托管长任务。
- 统一验证门禁。
- 自动保存 trace、日志、diff、截图、测试结果。
  4.2 第二层：Developer Feedback Loop
  这是人类判断循环，目标是让人从“盯实现细节”上移到“判断产品方向和工程质量”。
  输入：
- 当前可预览产品
- PR / diff / 截图 / 视频
- 任务完成证据
- 失败原因和风险点
- 用户目标和业务上下文
  人的工作不再是手动找每个小 bug，而是：
- 判断产品方向是否正确。
- 判断 UI / 交互 / 流程是否符合预期。
- 修改规格、约束、验收标准。
- 判断是否需要补 eval、补测试、补数据。
- 决定继续迭代、合并、上线或废弃。
  平台能力：
- 类 Orca 的本地界面：预览、diff、评论、测试、PR。
- 类 Multica 的任务界面：Agent 作为队友，汇报进展、阻塞和下一步。
- 规格更新入口：人的反馈直接写回任务、SPEC、验收标准和 Agent 上下文。
  4.3 第三层：External Feedback Loop
  这是外部反馈循环，目标是把真实世界的反馈自动变成下一轮产品判断。
  输入：
- 生产日志
- 监控告警
- 用户行为
- 业务单据
- 交易流水
- 审批记录
- 用户反馈
- A/B 实验结果
- 客户访谈和会议纪要
- 竞品变化
  执行过程：
1. 云端任务按 cron、Webhook 或业务事件触发。
2. Agent 拉取数据、清洗、汇总、归因。
3. 生成洞察：问题、机会、异常、趋势。
4. 形成产品假设、规格变更、bug、增长实验或重构任务。
5. 进入任务编排系统。
6. 触发 Developer feedback loop 或 Coding loop。
   平台能力：
- 持久任务与调度。
- 数据连接器。
- 浏览器自动化和 Python 分析环境。
- 人类审批等待点。
- 业务指标到研发任务的自动映射。
  这一层是平台最需要补齐的差异化能力。它决定平台是否只是“更好的 coding agent 管理器”，还是能真正形成持续迭代的软件工厂。

6. 本地 Agent 工作台与云端 Loop 控制台如何协作
   6.1 本地 Agent 工作台
   本地工作台是个人生产力界面。
   典型用户：
- Prototyper：快速试想法、开多个探索任务。
- Builder：把原型变成生产级实现。
- Sweeper：清理 UI、简化代码、修性能和质量问题。
- Maintainer：处理成熟系统的可靠性、安全、成本和效率问题。
  典型动作：
- 从云端任务打开本地 worktree。
- 启动一个或多个本地 Agent。
- 查看页面预览和 diff。
- 对某一行 diff 批注并发回 Agent。
- 本地验证通过后推送 PR。
- 把结果和证据同步回云端任务。
  6.2 云端 Loop 控制台
  云端控制台是组织级 loop 界面。
  典型用户：
- Grower：基于外部反馈持续优化 PMF。
- 产品负责人：审查方向、修改规格、决定下一轮实验。
- 技术负责人：控制质量门禁、风险和架构边界。
- 管理者：查看任务吞吐、阻塞、质量、成本和 Agent 参与度。
  典型动作：
- 从用户反馈或生产数据生成任务。
- 把任务分配给 Agent、Squad 或人。
- 查看任务状态和阻塞原因。
- 暂停任务等待人类审批。
- 查看 Agent 生成的分析、PR、截图、视频和验证报告。
- 决定合并、上线、继续迭代或废弃。
7. 面向未来角色的工作界面设计
   未来的工作界面不应再按“产品、研发、测试、运维、数据”这类职能岗位切分。AI 让不同职能的边界变薄，同一个人会在不同阶段承担不同工作模式：探索想法、构建产品、清理质量、推动增长、维护成熟系统。
   因此，平台界面应按 工作模式 组织。工作模式不是新的岗位名称，而是用户进入平台时选择的默认视图、默认信息密度和默认操作集合。

   Prototyper 视图强调多方案并行。用户输入一句话想法、草图、竞品链接或用户场景后，平台应能生成多个原型分支，并自动给出预览、差异说明、关键假设和待验证问题。这个视图的核心动作是比较、废弃、合并方向和转成正式 Spec。
   Builder 视图强调交付证据。Spec、任务拆解、代码变更、测试结果、页面预览和 PR 证据应在同一个界面里呈现。用户不需要盯每个 Agent 的执行细节，而是审查架构边界、权限变化、数据变更、质量门禁和发布风险。
   Sweeper 视图强调收敛质量。平台需要把删除、简化、合并、降复杂度作为一等任务，而不是只支持继续增加功能。每个清理任务都应该提供前后对比，包括代码量、截图、性能、复杂度、风险和回归验证。
   Grower 视图强调外部反馈。平台应聚合用户行为、业务指标、A/B 实验、客户反馈和竞品变化，由 Agent 生成洞察和产品假设，再由人决定进入研发、继续观察还是丢弃。这个视图连接的是 external feedback loop 和 developer feedback loop。
   Maintainer 视图强调运行治理。平台应接入监控、日志、告警、成本、依赖、漏洞和容量数据，让 Agent 自动分诊异常、定位原因、生成修复 PR 或运维建议。生产变更、权限、密钥、数据迁移和回滚必须进入人工审批。
   这些工作模式不应该变成五个割裂产品，而应是同一个平台里的五种工作视角。
   界面设计原则：
- 同一任务，不同视图：Prototyper 看想法和原型，Builder 看任务和 PR，Grower 看业务指标，Maintainer 看风险和运行状态。
- 默认信息密度不同：探索阶段看多方案，建设阶段看证据，增长阶段看指标，维护阶段看风险。
- 人只处理判断点：界面不应让人盯每个 Agent 动作，而应突出方向判断、质量门禁、风险审批和取舍。
- 所有反馈都能回写：评论、审批、废弃、合并、上线、指标变化都必须回写任务、Spec、业务世界模型或 Skill。
- 本地和云端共享任务状态：本地工作台适合高频操作，云端控制台适合编排、审查、审批和外部反馈。
8. 任务生命周期
   平台中的任务不应只是 Issue，也不应只是 Agent 会话。它应该是贯穿三层 loop 的工作单元。
   推荐状态：
1. Idea：来自人、外部反馈、数据异常或 Agent 发现。
2. Spec：被整理成规格、目标、边界和验收口径。
3. Ready：具备上下文、模板、环境和验证方式。
4. Running：Agent 正在本地或云端执行。
5. Needs Review：产出 PR、预览、截图、日志和验证证据。
6. Needs Human：需要人做方向、设计、权限或发布判断。
7. Merged / Shipped：合并或上线。
8. Observed：进入生产数据观察期。
9. Learned：反馈沉淀为模型、模板、Skill、eval 或新任务。
   关键原则：
- 工作围绕任务，而不是围绕会话。
- 一个任务可以产生多个 PR。
- 一个任务可以不产生代码，只产生调研、规格、实验设计或数据分析。
- Agent 可以创建后续任务，但排期和关键决策要可审计。
- 任务必须携带验证证据，而不是只带“已完成”的文本声明。
9. 业务世界模型
   业务世界模型是三层 loop 的共同上下文。
   它不只是需求文档，而是让 Agent 能够持续理解业务、系统和运行事实的资产。

   用户行为、业务单据、交易流水、审批记录、运营活动、异常事件、经营指标
   支撑外部反馈 loop 和生产数据回流
   工程知识
   模板、Skills、测试用例、集成适配器、质量门禁、部署配置、监控指标
   支撑 Agent 执行和验证
   产品判断
   用户画像、场景约束、设计原则、竞品信息、路线图
   支撑 developer feedback loop
   组织规则
   权限、审批、合规、发布窗口、风险等级
   支撑人类审批和治理
   平台真正的复利来自这里：
- 每次失败沉淀为 Skill、eval 或约束。
- 每次上线沉淀为生产事实。
- 每次人工反馈沉淀为规格和验收口径。
- 每次客户/用户反馈沉淀为产品判断。
10. 角色形态
    未来产品团队的角色会从职能分工转向能力组合。工程、产品、设计、数据科学的边界会继续融合，团队更需要围绕产品阶段配置不同能力角色。
    角色
    主要职责
    平台支持
    Prototyper
    提出新想法，快速生成多个原型
    本地工作台快速开任务、预览、废弃
    Builder
    把想法做成生产级产品/基础设施
    coding loop、模板、验证、PR
    Sweeper

11. MVP 建设路径
    阶段 1：本地工作台 + Coding Loop
    目标：让单个团队能稳定管理多个本地 Agent。
    最小能力：
- 任务列表。
- 本地 worktree 管理。
- 启动 Claude Code / Codex 等 CLI Agent。
- 查看终端、diff、预览、测试结果。
- 把结果回写任务。
  成功标准：
- 一个工程师可以稳定管理 3 到 5 个并行 Agent。
- 每个任务都有独立工作区和可追溯产出。
- 任务到 PR 的过程可复现、可审查。
  阶段 2：云端执行环境 + Developer Feedback Loop
  目标：把长任务和任务状态从本地搬到云端。
  最小能力：
- 云端任务队列。
- 长任务运行和重试。
- 并发控制。
- 人类审批等待点。
- 实时日志和状态。
- PR / 预览 / 验证证据聚合。
  成功标准：
- 人不需要盯终端，也能知道每个 Agent 在做什么。
- 任务失败能看到失败原因并可重试。
- 人的反馈能写回规格和下一轮任务。
  阶段 3：External Feedback Loop 自动化
  目标：让上线后的业务事实自动进入研发系统。
  最小能力：
- 接入日志、监控、用户行为、业务数据和反馈渠道。
- 定时分析和异常检测。
- 生成洞察、bug、实验建议或产品假设。
- 支持人类审批后进入任务系统。
  成功标准：
- 生产数据能自动生成可审查的迭代建议。
- 业务异常到定位结论的时间显著缩短。
- 外部反馈能稳定进入任务编排，而不是停留在会议纪要。
  阶段 4：多 Agent 编排与生产线模板
  目标：从“多个 Agent 并行”升级为“软件生产线”。
  最小能力：
- 任务依赖 DAG。
- Agent Squad。
- 模板化生产线。
- 跨任务学习：失败模式回流为 Skill / eval / 模板约束。
- 组织级指标看板。
  成功标准：
- 高频业务应用可以通过模板快速生产。
- 同一生产线适配新业务的周期持续下降。
- Agent 执行、人工反馈、外部数据形成闭环复利。
12. 核心指标
    指标类别
    指标
    Coding loop 效率
    需求到 PR 时间、需求到预览环境时间、Agent 独立完成任务占比、CI 一次通过率、自动修复成功率
    Developer feedback 效率
    人审等待时间、反馈到规格更新时间、反馈到下一轮任务时间、审查包完整率
    External feedback 效率
    生产数据接入覆盖率、异常到定位时间、异常到修复 PR 时间、数据触发迭代数量
    编排能力
    并行 Agent 数、任务吞吐、卡住任务恢复率、任务依赖自动展开率、失败重试成功率
    生产线能力

    治理能力

13. 对标启发
- Orca：证明本地 Agent 工作台需要围绕多 Agent、worktree、预览、diff review 和真实 CLI Agent 来设计，而不是在传统 IDE 里加聊天框。
- Trigger.dev：证明长程 Agent 和外部反馈 loop 需要 durable task、队列、并发控制、调度、人类审批、浏览器和 Python 运行能力。
- OpenAI Symphony：证明任务追踪器可以成为 Agent 控制面。关键不是管理会话，而是保证每个待办任务都有 Agent 在独立工作空间中持续推进。
- Multica：证明 Agent 可以被当作团队成员管理。任务、状态、阻塞、技能复用和 Squad 路由，会成为组织级 Agent 平台的基本形态。
- Loop Engineering：提醒我们平台的核心不是 prompt，而是目标、上下文、工具、状态、验证、升级和停止条件组成的运行回路。
14. 最终定位
    智能研发平台的目标不是替代工程师，也不是把所有研发工作自动化。
    它要做的是把三件事产品化：
1. 把 coding loop 自动化：让 Agent 在明确边界和验证条件下持续执行。
2. 给 developer feedback loop 建工作界面：让人高效判断方向、修改规格、确认质量。
3. 把 external feedback loop 接入研发系统：让生产数据、用户反馈和业务信号自动进入下一轮迭代。
   最终形态是：
   一个面向业务型应用软件的 AI Loop Platform：本地工作台承载人与 Agent 的高频协作，云端执行环境负责长程任务和外部反馈自动化，任务编排系统负责让多 Agent 像团队一样持续推进软件生产。
