# Raven 产品定义（v1 基准）

> 本文档是所有设计讨论的收敛基准。术语以 [/CONTEXT.md](../CONTEXT.md) 为准，硬决策以 [docs/adr/](./adr/) 为准。与本文档冲突的旧材料（调研文档、智能化平台.md）以本文档为准。
> 最后更新：2026-07-04

## 1. 定位

**企业级智能研发执行平台的控制面。** 以需求表达为起点，把需求转化为可执行、可追踪、可审计的交付过程，由外部 coding agent 在受控 workflow 中完成闭环。

三层分离：

- **意图层**：人提出需求（v1 用户 = 工程师/技术负责人，业务自然语言直达是 v2+ 北极星）
- **编排层**：workflow 把需求翻译成受控交付过程——平台本体，核心差异点
- **执行层**：multica（派单/看板/runtime）+ trigger.dev(持久运行) + CLI agent（可替换，不自研）

核心论点（对外一句话）：**沉淀复利的不只是 skill，更是 workflow——组织的交付判断力被编码为可执行、可版本化、可度量的资产。**

## 2. 与对标项目的关系

| 项目 | 关系 |

## 3. 需求生命周期

九状态（schema 全量定义，见 CONTEXT.md）：

```
Idea → Spec → Ready → Running → Needs Review → Needs Human → Merged/Shipped → Observed → Learned
```

- v1 活跃路径到 **Merged**（PR 合入，GitHub webhook 或人工确认触发）；Observed/Learned 状态存在、无自动化，Learned 可手动标记
- 状态机由控制面自持，multica issue 状态是单向投影
- 需求澄清不做独立 UI：workflow 第一节点 = agent 读库做功课后带方案回 issue 评论区，问 2-3 个拍板问题，产出结构化 Spec，过 Ready 门禁
- 原则：任务必须携带验证证据，而不是只带"已完成"的文本声明

## 4. Workflow 体系

- **SDK**（ADR-0005）：`defineWorkflow({contract}, script)`，受控原语 `agent() / gate() / evidence() / squad()`，合同强制声明。人审批合同，机器执行脚本
- **部署**：git-backed（ADR-0002），改动走 commit + review + Actions deploy
- **三条产生路径**：人手写；Squad 运行轨迹提炼（孵化器，ADR-0003）；自然语言描述 → agent 生成草稿。全部经人审注册
- **失败归属**：重试/超时/换 agent 策略只属于 workflow 声明，执行层各层只如实上报
- **v1 数量**：平台团队手写 ≤3 条（中小功能交付、bug 修复），不开放编写入口
- **信任分级**：v1 只埋数据（每个门禁记录通过/驳回/原因），降级机制 v2 上。挣足信任的 workflow 称"生产线"

## 5. 治理

- **证据链**：结构化证据存控制面 Postgres + 对象存储，multica 评论只放摘要+回链；v1 证据源 = GitHub PR/Actions（diff、CI 状态、测试输出）
- **审批**：v1 全部在平台内完成（审查包页面），飞书暂缓
- **审计**：控制面事件表（用户操作、agent 动作、门禁裁决）+ trigger.dev run 历史
- **Agent 权限**：粗粒度在平台（workflow 合同声明），细粒度落在 harness 配置层（Claude Code settings / sandbox），平台负责集中管理与下发

## 6. v1 范围

**做**：控制面模块（状态机/证据/审计，multica server 内新增）、Raven SDK、审查包 UI（apps/web 内新增页面）、trigger.dev 自托管、2-3 条 workflow、GitHub App/webhook 集成。

**明确不做**：自建本地工作台（工程师用自己的终端+IDE）、External loop 数据分析（只留 webhook→Idea 入口）、五种工作模式视图（只做 Builder 视角，视图从 workflow 合同生成）、业务世界模型（v1 = 仓库文档+workspace context+skill）、部署与发布阶段（止步 PR）、飞书集成、信任降级机制、workflow 编写开放。

## 7. 环境与运行形态

- 模型：集团内可直接使用 Claude/OpenAI，agent 预算无限
- 代码托管/CI：GitHub + Actions，交付止步于 PR
- 部署：无 k8s，v1 全套（multica、trigger.dev、控制面）本机/单机 Docker 跑
- 团队：两人全职（jjyang + Claude）
- 试点：**平台自举**——用 Raven 交付 Raven 自己的需求；成功定义 = 第一条 workflow 完整交付一个真实需求 PR

## 8. 技术栈与仓库

- 唯一仓库 = multica fork（LinkDashSaber/multica），Raven 在其中以新模块生长（ADR-0001）
- 技术栈跟随宿主（ADR-0004）：server 侧 Go（状态机/证据/审计）、UI 侧 apps/web 新页面、workflow 侧新 TS 包（Raven SDK + trigger.dev）
- 工程纪律：Raven 迁移用 900+ 编号段；少碰路由注册/侧边栏/lockfile 等冲突高发文件；上游经 `git merge upstream/main` 接收

## 9. 里程碑

1. **M1 骨架**：Raven 模块脚手架（server 模块 + 900 段迁移 + workflow TS 包）、状态机 + 证据 schema、trigger.dev 自托管跑通、workflow→派单链路打通
2. **M2 第一条 workflow**：`feature-delivery` 走通 澄清→计划→执行→自验→PR→人审 全链，审查包 UI 最小版
3. **M3 自举达成**：Raven 的真实需求通过 Raven 交付（成功标准），门禁数据开始积累
4. **M4 扩展**：第二三条 workflow、Squad 孵化管道、workflow 度量看板

## 10. 核心指标（自举期）

需求到 PR 时间、CI 一次通过率、门禁人工驳回率及原因分布、审查包完整率、agent 独立完成占比、每需求 token 成本。（完整指标体系见 智能化平台.md 第 12 节，随载体成熟逐步启用）
